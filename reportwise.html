The Problem

Lab reports are written for doctors, not for the people they belong to

When most people receive a blood test report, they see a wall of acronyms (LDL, HbA1c, TSH), numbers with units, and reference ranges in tiny print. The same value that says "high" on one report says "borderline" on another lab. Patients often wait days for their doctor to translate what is otherwise unreadable.

Core Insight
The hardest part was not the AI — it was deciding what the product would refuse to do. No diagnosis. No medicine recommendations. No symptom matching. The product explains; it does not interpret.
Product Hypothesis

If we can extract structured data from any lab PDF, plain-language explanations can be safe and useful

Multimodal AI is now strong enough to read varied PDF layouts and return clean structured data. If that piece works reliably, the rest of the product — flagging, explanations, multilingual UI, comparison — becomes a thin layer on top. The hypothesis was that this layer, kept small and clearly bounded, could be genuinely useful without straying into clinical advice.

Target User

Adults trying to understand a lab report for themselves or a family member

A non-clinical reader who:

  • Just received a routine blood test, annual check-up or pre-employment screening
  • Wants to know what is worth raising at the next doctor visit, not a diagnosis
  • Often reads reports on behalf of a parent or older family member
  • Prefers a regional language for explanations even if the lab report itself is in English
  • May have an older report to compare against

The user picks a PDF, solves an invisible Cloudflare bot challenge (usually no interaction needed), and the server validates the file before any AI call. Markers are extracted as structured JSON, "high/low/normal" is computed in the app itself against the printed range, and explanations are layered in afterwards in batches.

Key Product Decisions

Seven calls that shaped the product

1. Use the report's own reference range, never an external one
Labs in different cities use different ranges. A value of 49 mg/dL is "low" on one report and "normal" on another. ReportWise uses only what is printed on the uploaded PDF — this avoids any pretence of clinical judgement and keeps the product honest.
2. Explain, do not interpret
Every explanation answers "what does this test measure?" — never "what disease does this point to?". The system prompt forbids diagnoses, disease names, and medicine or supplement advice.
3. Triage UI: flagged first, normal collapsed
Out-of-range markers appear as full cards at the top. Normal values are grouped into health-area panels that stay collapsed until tapped. The user's attention is steered to what is worth a conversation with their doctor.
4. Compute high/low/normal in app code, not in the AI response
The AI returns the raw value, unit and printed range. The app does the arithmetic and the classification. This removes a class of "the AI got the verdict wrong" bugs and makes the logic auditable.
5. Five languages, but marker names stay in English
Test names like "Hemoglobin", units like "mg/dL", and reference ranges are never translated — they must match the printed report. Only labels, explanations, doctor questions and the safety note translate.
6. Detect duplicate uploads before Compare runs
If the user uploads the same PDF twice into Compare, the app detects it after extraction (same date + ≥85% identical values) and warns them instead of producing a useless "no change" dashboard.
7. No accounts, no storage, no retention
The product never persists the file, the extracted markers, or the patient name. Each session is independent. This kept the privacy story simple enough to explain in one line on the upload screen.
Architecture

A static front end and a single serverless function

The whole product runs on Vercel's free tier. The browser holds the UI; one serverless function holds every secret and gate.

Browser
Static HTML, CSS and vanilla JS. Renders the upload screen, dashboard, comparison view, and the Cloudflare Turnstile widget. Never sees any API key.
Edge headers
CSP, HSTS, frame-ancestors, Permissions-Policy. Locked down so only Cloudflare's iframe, Google Fonts and our own origin can load resources.
Serverless API
One function: /api/messages. Runs every gate in sequence: kill switch → origin allowlist → Content-Type → Turnstile verification → PDF validation → rate-limit check → AI call → response sanitisation.
Bot verification
Cloudflare Turnstile. Verified server-side before any quota is consumed. Free, invisible for trusted users.
Rate limiting
Upstash Redis (free tier). Atomic Lua scripts handle four counters, an active-request lock and an idempotency key in one round trip. Roughly two Redis commands per analysis.
AI
Google Gemini (multimodal). Reads the PDF directly, returns structured JSON. Falls back from Flash Lite to Flash automatically when the cheaper model is overloaded.
Storage
None. No database. No object storage. Files exist in memory for the duration of one request.
Security and Privacy

Defence in layers

Health-adjacent products attract abuse: bots probing for API keys, attackers uploading malicious files, prompt-injection attempts hidden inside PDF text. The product treats every layer as suspect and validates accordingly.

🔒 API key protection
All AI calls go through a server function. The Gemini key lives only in Vercel's environment variables and is never sent to the browser.
📄 Hard PDF validation
Server checks %PDF magic bytes, EOF marker, file size, page count and refuses encrypted PDFs — before any AI call.
⚡ Rate limits
2 per 15 min, 5 per hour, 10 per day per IP. Compare counts as 2 credits. A global daily cap protects against runaway cost.
🤖 Bot verification
Cloudflare Turnstile verified server-side before quota is consumed. Invisible for trusted users, blocks scripted abuse.
🧱 Prompt-injection shield
A security preamble is prepended to every PDF call telling the model to treat PDF contents as untrusted data, never instructions.
🧼 Response sanitisation
The AI's reply is parsed against a strict schema. HTML tags, control characters and unknown status values are stripped before render.
🛑 Kill switch
A single environment variable disables analysis instantly without redeploying the front end.
🔍 Safe logging
No PDF contents, patient data, marker values or full AI responses ever reach the logs. IPs are hashed before storage.
Fail-Closed in Production
If Upstash or Turnstile environment variables are missing in production, the API refuses every request with a 503 instead of silently allowing unprotected traffic. The opposite default of most deployments.

Caregiver mode. Many users read reports for an elderly parent. A simple two-profile setup (you and one elder) would match how the product is actually used.
Key Learnings

What this project taught me

A
Scope discipline is a product skill. Saying no to "but what if it also suggests medicine" was harder than saying yes. Every refused feature made the security and safety story possible.
B
Treat the AI as one component, not the product. Building reliable validation around the model — both before and after — turned a fragile prototype into something that fails gracefully when the AI fails.
C
Free-tier limits force good architecture. Designing around Upstash's 10k commands/day budget pushed the rate-limit logic into a single Lua script — cleaner than the naive multi-command version.
D
Fail-closed is a product decision. Defaulting to "block the request" when a config is missing protects users at the cost of making the operator (me) more careful at deploy. Right trade-off for a health tool.
