The Problem

Households need a simple way to track shared meals and nutrition

Most food tracking apps are single-user and require tedious manual data entry. Families and roommates who cook together want to log meals collaboratively, see who ate what, and understand the household's overall nutrition patterns without friction.

❌ Current Apps
Single-user only. No way to share meals or track household patterns
Manual data entry. Searching and logging every item is tedious
No collaboration. Can't invite roommates or family members
Limited databases. Many missing foods or inaccurate nutrition data
✓ HomePlate
Household accounts. One household, multiple users with invite codes
Fast search. 1000+ foods with real nutrition data; quick add-to-meal flow
Real-time sync. Everyone sees meals instantly; daily summaries update live
Comprehensive database. Common foods, branded items, and custom food support
Core Insight
Building for households, not individuals, unlocks collaboration. When people log food with roommates, they're motivated to be more consistent because meals are shared and visible to the group.

A household-first design with a rich food database removes friction from daily logging

By combining multi-user authentication, a curated database of 1000+ foods with accurate macros, and an invite-based household system, we can make food tracking a household habit instead of a solo chore. Real-time updates and mobile-optimized design keep the barrier to entry low.

Target User

Households and roommates who cook and eat together

The primary user is someone who:

  • Lives with 1–4 other people (roommates, partner, family)
  • Cooks shared meals or wants to track collective nutrition
  • Wants accountability through group visibility
  • Uses mobile as the primary device during meals
  • Values speed over exhaustive data entry

After signup, the user either creates a household (getting a unique 6-char join code) or joins an existing one. Once in, they land on the daily summary page showing the current day's meals. A fast search interface lets them find and add foods to any meal. All data syncs in real-time with Supabase, and household members see each other's logs immediately.

Key Product Decisions

Five architectural decisions that shaped the product

1. Household model with invite codes instead of friend requests
Roommates and families want frictionless collaboration. A 6-character code is faster than managing friend requests. One household per user simplifies permissions and data access.
2. Curated database of 1000+ foods instead of free-form entries
User-created foods lead to duplicates and inconsistent nutrition data. A structured food database with standard macros ensures reliable summaries across the household.
3. Real-time Supabase sync instead of cached local state
When roommates are logging at the same time, real-time updates matter. Using Supabase RLS policies and listeners ensures every client sees consistent, up-to-date data.
4. Row-Level Security (RLS) for data isolation
Every query is filtered by `user_id` and `household_id` at the database level, not in application code. This prevents data leaks and simplifies the trust model.
5. Mobile-first UI with React + Tailwind
Logging happens on phones during meals. The entire UI is designed for thumb-friendly interaction and fast load times, with desktop as a secondary experience.
Architecture

Next.js client with Supabase backend

A straightforward full-stack architecture: React components on the front end, Supabase for authentication and realtime data, and Vercel for hosting.

Browser
Next.js app with React and Tailwind CSS. Renders the login, household setup, daily logger, and summary views. Client-side state with Zustand store.
Auth
Supabase email/password auth. Sessions stored in localStorage and synced across browser tabs. onAuthStateChange listener keeps the Zustand store in sync.
Realtime Data
Supabase Postgres with Row-Level Security. Every INSERT/UPDATE/DELETE is filtered by user_id and household_id at the database layer. Listeners broadcast changes in real-time to all connected clients.
Food Database
Pre-populated 1000+ food entries. Each food record includes name, serving size, and macro/micronutrient data. Users log via search + quick-add.
Hosting
Vercel for the Next.js app. Supabase hosts the database. Automatic deployments on git push.

Key Learnings

What built this product

1
Household model over peer-to-peer. Treating the household as the unit of collaboration simplified permissions and data ownership.
2
Real-time sync is essential for shared apps. Without instant updates, logging together feels broken. Supabase's realtime listeners made this frictionless.
3
Data quality > quantity. A curated 1000-food database is more useful than 10,000 foods with missing macros.
4
RLS at the database layer, not the app. Filtering data in SQL ensures no query bypass bugs. Trust the database, not the app.
5
Mobile-first from day one. Every component was sized and spaced for thumb interaction. Desktop felt like a natural extension.
6
Session persistence matters. Storing auth tokens in localStorage and syncing across tabs kept users logged in across app restarts.
Next Steps
Future iterations could add barcode scanning for instant food lookup, a recipe builder for shared meals, and integrations with fitness trackers. The foundation is solid for multi-user household nutrition.
