The Problem

Existing calorie trackers don't fit homemade Indian meals

When my husband and I wanted to track nutrition consistently, every existing app made it inconvenient:

❌ Existing Trackers
Individual-first. Solo users, no household sharing
Generic portions. Cups, ounces, grams — not katoris
Missing regional food. Tomato dal, bottle gourd curry not in database
Duplicate logging. If we eat the same meal, we log it twice
Invisible preparation. Oil level, added ingredients ignored
✓ HomePlate
Household-first. Two members, shared data, separate goals
Food-specific units. Katori for curry, pieces for eggs, scoops for protein
Homemade focus. Regional South Indian meals and local varieties
Log once, portion twice. Configure dish, assign each person's quantity
Preparation options. Oil level (85%–118%), optional additions
Core Insight
The challenge was not only finding calorie values. It was representing how food is prepared, measured and shared at home.

Couples eating homemade meals together with different goals

Product Decisions

Seven deliberate choices that shaped the product

1. Configurable household profiles
Instead of a solo tracker, every user starts by creating a household with two members. Each member has a name, avatar color and separate nutrition goals. This makes sharing the primary model, not an afterthought.
2. Shared dish with separate portions
When logging a meal, users select "Both" and assign different quantities to each person. This creates two diary entries linked to the same source, avoiding duplicate work while preserving individual nutrition tracking.
3. Food-specific serving units
Instead of forcing all foods into grams or cups, each food defines which units make sense: curries use katori/cup/gram; eggs use pieces; protein powder uses scoops/grams. This eliminates mental math and makes logging faster.
4. Approximate estimates with transparent labels
Homemade food values are labeled as "approximate" with source notes (NIN India, USDA, recipe estimation). Users understand the values are reasonable approximations, not precise measurements. This builds trust without false precision.
5. Manual food creation over comprehensive database
Instead of trying to build a 100K-food database, users can quickly create custom foods or quick-enter just calories. This avoids the friction of searching and reduces FOMO about missing foods.
6. Cloud-backed shared data via Supabase
Household data lives on the cloud. Both members can access the same diary from different devices. No syncing issues, no local-storage conflicts, no data loss if a device is lost.
7. Authentication and household privacy
Only household members can see household data, enforced by Row-Level Security (RLS) at the database level. This is not optional; it's built into every query.
Serving Units Architecture

The complexity of representing how food is measured at home

Different foods are measured in different units. The app supports this by storing each food's allowed units and gram equivalent:

Food Category Units Notes
Curries & dal katori, cup, gram 1 katori ≈ 150g
Eggs, dosa, idli pieces 1 piece varies
Drinks & soups ml, cup 1 cup ≈ 240ml
Meat & fish gram, piece 100g breast ≈ 1 piece
Bread & toast slice 1 slice ≈ 30g
Protein powder scoop, gram 1 scoop ≈ 30g (varies)
Fruits piece, cup, gram Food-specific

This flexibility is essential. Without it, users either guess or abandon tracking.

Key Learnings

What this project taught me

1
Workflow differentiation matters more than feature novelty. Shared meal logging didn't require a new database structure—just a UI convention. But it solved a real pain point.
2
Data architecture directly affects UX. Immutable diary snapshots (storing complete nutrition at logging time) prevented confusion about historical accuracy. Trade-off: simpler code, not perfect normalization.
3
Transparent approximation builds trust. Labeling values as "approximate" with source notes is more useful than false precision. Users understand the limitations.
4
Authentication and authorisation are separate requirements. Supabase Auth handles "who are you"; RLS handles "what data can you see." Don't conflate them.
